# Основные классы IFC для BIM-координатора АР

Новичка легко испугать длинными списками `IFC`-классов. На старте это действительно не нужно. BIM-координатору АР важнее не заучить весь словарь стандарта, а собрать рабочий набор сущностей, через которые обычно читается архитектурная модель.

То есть цель этой главы — не энциклопедия, а спокойная стартовая карта.

## Зачем вообще нужны классы

Для внешней системы есть большая разница между:

- зданием и этажом;
- стеной и произвольной геометрией;
- окном и абстрактной вставкой;
- пространством и просто пустым объемом.

Класс в `IFC` как раз и помогает зафиксировать этот смысл. Без него модель начинает хуже читаться, хуже фильтроваться и хуже проверяться.

## Стартовый набор для архитектурного чтения

Полезно собирать базовые классы не алфавитом, а по ролям в модели:

| Группа | Класс | Зачем важен |
| --- | --- | --- |
| Контекст проекта | `IfcProject` | задает верхний уровень проектной структуры |
| Контекст объекта | `IfcSite` | связывает модель с участком или площадкой |
| Объект как целое | `IfcBuilding` | помогает читать здание как единую сущность |
| Этажность | `IfcBuildingStorey` | критичен для читаемости этажей, уровней и пространственной структуры |
| Архитектурные элементы | `IfcWall` | позволяет системе понимать стену как стену, а не как произвольное тело |
| Архитектурные элементы | `IfcSlab` | важен для перекрытий и части логики этажей |
| Архитектурные элементы | `IfcDoor` | помогает корректно читать проемную логику |
| Архитектурные элементы | `IfcWindow` | важен для фасадного и архитектурного чтения |
| Пространственная логика | `IfcSpace` | один из ключевых классов для помещений, зон, площадей и дальнейших проверок |
| Сигнал риска | `IfcBuildingElementProxy` | допустим как исключение, но массовое появление обычно означает слабую классификацию |

Этого набора уже достаточно, чтобы у новичка появилась первая внутренняя карта: `IFC` — это не страшный каталог аббревиатур, а понятный язык, которым модель описывает свои основные сущности.

## Почему для АР особенно важен IfcSpace

Среди стартовых классов для архитектурного координатора особенно выделяется `IfcSpace`. Именно через него тема пространств, помещений, зон и расчетной логики начинает связываться с обменной моделью.

Позже это станет особенно важным в модулях про:

- `АГР`;
- площади и `ТЭП`;
- проверку пространственной логики;
- замечания по читаемости архитектурной модели.

Поэтому `IfcSpace` полезно с самого начала воспринимать не как “еще один технический класс”, а как один из центральных рабочих мостов между архитектурной моделью и внешним чтением.

## Как не ошибиться в изучении классов

На старте полезно держаться трех правил:

1. Не пытаться выучить весь IFC-словарь сразу.
2. Смотреть на класс как на смысловую роль объекта, а не просто на название.
3. Всегда спрашивать, зачем этот класс важен для чтения, проверки или передачи модели.

Это заметно полезнее, чем бессистемное запоминание длинных перечней.

## Что дает это координатору на практике

Когда координатор знает базовые классы архитектурного контура, ему легче:

- замечать неправильную или слишком грубую классификацию;
- раньше видеть, когда модель уходит в `IfcBuildingElementProxy` вместо осмысленных сущностей;
- понимать, почему система “не видит” объект так, как ожидалось;
- разбирать экспортные проблемы не как хаос, а как конкретный сбой смыслового чтения;
- связывать архитектурную модель с ее будущим использованием во внешнем контуре.

## Рабочий вывод

`IFC`-классы нужны не для усложнения жизни, а для того, чтобы модель читалась как система осмысленных сущностей. Для BIM-координатора АР на старте достаточно уверенно держать базовый набор: контекст проекта, этажность, ключевые архитектурные элементы и особенно пространства. Этой опоры уже хватает, чтобы перестать смотреть на `IFC` как на черный ящик.
