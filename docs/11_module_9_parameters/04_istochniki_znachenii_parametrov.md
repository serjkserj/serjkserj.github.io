# Источники значений параметров

## О чем эта глава

Одна из самых важных и недооцененных тем в параметризации — не список полей, а источник их значений.

Новичок часто думает так: если параметр существует, значит его просто кто-то заполнит. На практике именно здесь рождается половина хаоса.

## Простое объяснение темы

Источник значения параметра — это ответ на вопрос, откуда берется конкретная информация и почему ей можно доверять.

Для одного поля источником может быть задание на проектирование, для другого — расчет, для третьего — геометрия модели, для четвертого — решение архитектора, а для пятого — согласованный классификатор.

Пока источник не определен, параметр остается сомнительным, даже если в нем уже что-то написано.

## Почему это особенно важно

Если у параметра нет понятного источника, возникают типовые проблемы:

- значение вводится "на глаз";
- разные участники проекта пишут разное;
- поле невозможно проверить;
- при обновлении модели данные быстро устаревают;
- никто не понимает, кто отвечает за исправление.

Именно поэтому зрелая параметризация начинается не с вопроса "какие поля добавить", а с вопроса "откуда в эти поля должны приходить значения".

## Какие источники встречаются чаще всего

В реальной работе значения параметров обычно приходят из нескольких типов источников:

- исходно-разрешительная и проектная документация;
- нормативные и методические требования;
- расчетные таблицы и показатели;
- сама геометрия модели;
- уровни, помещения, зоны и их взаимосвязи;
- согласованные классификаторы и справочники;
- решения профильного проектировщика;
- внутренние проектные договоренности.

Последний пункт особенно важен: внутренние договоренности могут быть полезны, но их нельзя автоматически превращать в универсальную норму для всех проектов.

## Схема

Удобно видеть это не как список полей, а как маршрут происхождения значения:

```mermaid
flowchart LR
    A[Документ / расчет / геометрия / справочник] --> B[Источник значения]
    B --> C[Параметр в модели]
    C --> D[Владелец параметра]
    C --> E[Проверка]
    E --> F[IFC / таблица / проверка / выдача]
```

## Где это встречается в проекте

Для BIM-координатора тема источника значений всплывает постоянно:

- когда нужно понять, кто должен заполнить параметр;
- когда модель не проходит проверку по данным;
- когда один и тот же атрибут по-разному понимают разные специалисты;
- когда требуется собирать показатели из модели и объяснять их происхождение;
- когда в проекте появляются внутренние поля, которые живут отдельно от официальных требований.

## Что должен делать BIM-координатор

Координатору полезно задавать по каждому важному параметру пять рабочих вопросов:

1. зачем этот параметр нужен;
2. кто его владелец;
3. откуда берется значение;
4. в какой момент оно должно быть заполнено;
5. как это значение проверить.

Если хотя бы на один из этих вопросов нет ответа, параметр уже находится в зоне риска.

## Типовые ошибки новичков

- Считать, что главный вопрос — наличие параметра, а не происхождение его значения.
- Допускать ручной ввод там, где значение должно определяться из более надежного источника.
- Не различать обязательные требования и внутренние удобные поля команды.
- Не назначать ответственного за заполнение и актуализацию.

## Короткий вывод

Хороший параметр — это не просто ячейка с текстом или числом. Это параметр с понятным смыслом, источником, владельцем и способом проверки.

Для BIM-координатора это один из главных признаков зрелой модели: данные в ней должны быть не только заполнены, но и объяснимы.

Эта глава напрямую связывает модуль про параметры с модулем про показатели и с будущим `QC`: как только у значения нет понятного источника, у модели почти сразу появляется проверочный риск.
